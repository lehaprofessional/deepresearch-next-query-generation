## 5. Дообучение локальной модели методом LoRA

### 5.1 Цель эксперимента

После построения prompt-based baseline был проведён эксперимент по дообучению локальной генеративной модели для задачи генерации следующего поискового запроса.

Постановка задачи:

```text
research_question
+ previous_queries
+ descriptions of visited websites
→ next_search_query
```

Цель эксперимента — проверить, может ли LoRA-адаптация на небольшом наборе реальных примеров улучшить базовую локальную модель.

### 5.2 Датасет

Для supervised fine-tuning использовались реальные примеры из Task 4, дополненные описаниями посещённых источников из Task 3.

Параметры датасета:

| Split | Examples | Research questions |
|---|---:|---:|
| Train | 248 | 14 |
| Validation | 32 | 3 |

Разделение выполнено по `research_question`, поэтому одинаковые исследовательские вопросы не пересекаются между train и validation.

Дополнительно было проверено:

- research-question leakage: 0;
- одинаковых пар input–target между train и validation: 0;
- средняя длина train-примера: 1277.9 токена;
- максимальная длина train-примера: 2028 токенов;
- все примеры помещаются в ограничение `2048` токенов.

Validation-набор оставался полностью реальным и не использовался для обучения.

### 5.3 Конфигурация обучения

В качестве базовой модели использовалась:

```text
Qwen/Qwen2.5-1.5B-Instruct
```

Метод адаптации:

```text
LoRA — Low-Rank Adaptation
```

Основные параметры:

| Parameter | Value |
|---|---:|
| LoRA rank | 16 |
| LoRA alpha | 32 |
| LoRA dropout | 0.05 |
| Learning rate | 1e-4 |
| Effective batch size | 8 |
| Maximum sequence length | 2048 |
| Precision | BF16 |
| Requested epochs | 5 |
| Early stopping patience | 2 |

LoRA применялась к attention- и MLP-проекциям:

```text
q_proj, k_proj, v_proj, o_proj,
gate_proj, up_proj, down_proj
```

Количество обучаемых параметров:

```text
18,464,768
```

Это составляет примерно:

```text
1.182% от общего числа параметров модели
```

Обучение проводилось на видеокарте:

```text
NVIDIA GeForce RTX 5070 Ti, 16 GB VRAM
```

Пиковое потребление видеопамяти составило около `4.82 GB`.

Обучение заняло приблизительно `6 минут 44 секунды`.

### 5.4 Поведение validation loss

Validation loss по эпохам:

| Epoch | Eval loss |
|---:|---:|
| 1 | 3.312 |
| 2 | **3.195** |
| 3 | 3.279 |
| 4 | 3.405 |

Лучший результат был получен после второй эпохи:

```text
checkpoint-62
eval_loss = 3.1949
```

После второй эпохи train loss продолжал уменьшаться, однако validation loss начал расти. Это свидетельствует о начале переобучения на небольшом train-наборе.

Early stopping завершил обучение после четвёртой эпохи и восстановил лучший checkpoint.

### 5.5 Сравнение базовой модели и LoRA

Генерация была выполнена на одинаковых 32 validation-примерах.

| Model | Token F1 | Jaccard | BERTScore F1 | Avg. query length |
|---|---:|---:|---:|---:|
| Qwen2.5-1.5B base | 0.2092 | 0.1256 | 0.8433 | 17.03 |
| Qwen2.5-1.5B + real-only LoRA | **0.2199** | **0.1346** | **0.8522** | **11.41** |
| Target queries | — | — | — | 10.94 |

Относительное улучшение LoRA по сравнению с базовой Qwen:

- Token F1: примерно `+5.1%`;
- Jaccard: примерно `+7.2%`;
- BERTScore F1: примерно `+1.1%`.

BERTScore подтвердил, что после LoRA семантическая близость к эталонным запросам увеличилась с `0.8433` до `0.8522`.

### 5.6 Качественные изменения

Базовая Qwen часто копировала формулировки из инструкции:

```text
mechanism + material + process condition
```

Также она генерировала слишком длинные запросы: средняя длина составляла `17.03` токена при эталонном значении `10.94`.

После LoRA запросы стали:

- короче;
- ближе к стилю технической поисковой строки;
- конкретнее по материалам и параметрам процесса;
- ближе к эталонным запросам как лексически, так и семантически.

Средняя длина после обучения составила `11.41` токена, что практически совпадает с эталонной длиной.

При этом сохранились отдельные ошибки:

- лишние кавычки внутри запросов;
- повторение популярных шаблонов, например `lance height`;
- генерация неподтверждённых численных параметров;
- выбор тематически связанного, но не всегда правильного следующего аспекта поиска.

### 5.7 Сравнение с prompt-based подходами

| Approach | Token F1 | Jaccard | BERTScore F1 |
|---|---:|---:|---:|
| Qwen2.5-1.5B base | 0.2092 | 0.1256 | 0.8433 |
| Qwen2.5-1.5B + LoRA | 0.2199 | 0.1346 | 0.8522 |
| Context prompt v2 | **0.2784** | **0.1753** | 0.8606 |
| Hybrid router | 0.2998 | 0.1917 | **0.8638** |

LoRA улучшила локальную модель по всем метрикам, однако пока не достигла качества более крупной внешней LLM с context-aware prompt engineering.

Hybrid router остаётся exploratory upper-bound, поскольку правила выбора стратегии были сформулированы после анализа validation-результатов.

### 5.8 Вывод

Эксперимент подтвердил, что LoRA-дообучение работает даже на небольшом датасете из 248 реальных примеров.

Основной результат:

> Real-only LoRA улучшила Qwen2.5-1.5B по Token F1, Jaccard и BERTScore, а также научила модель генерировать запросы подходящей длины и формата.

При этом небольшой объём и ограниченное тематическое разнообразие данных пока не позволяют локальной модели превзойти сильный prompt-based baseline.

Следующим экспериментом может стать обучение на смеси реальных и отфильтрованных синтетических данных. Validation-набор при этом должен оставаться неизменным и полностью реальным.
