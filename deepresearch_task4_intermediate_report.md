# Промежуточный отчёт по задаче 4

## Постановка задачи

На текущем этапе была реализована упрощённая версия задачи генерации поисковых запросов для DeepResearch-агента. Так как файл `train_summerschool_task3.csv` с контекстом посещённых сайтов пока отсутствует/пустой, задача была сведена к постановке:

```text
research_question + previous_queries → next_query
```

То есть модель генерирует следующий поисковый запрос на основе исходного исследовательского вопроса и уже сгенерированных ранее запросов.

После предоставления непустого `task3` планируется перейти к полной постановке задачи:

```text
research_question + visited_sources/descriptions → next_query
```

Это позволит учитывать контекст уже посещённых сайтов и более корректно моделировать поведение DeepResearch-агента.

---

## Использованные метрики

Для оценки качества использовались лексические метрики:

- **Exact Match** — полное совпадение с эталонным запросом.
- **Token F1** — частичное совпадение по словам с учётом precision и recall.
- **Jaccard** — доля общих уникальных слов между сгенерированным и эталонным запросом.
- **Repeated previous query** — проверка, не повторяет ли модель предыдущие запросы.

Exact Match оказался слишком строгой метрикой для задачи генерации поисковых запросов, так как один и тот же поисковый смысл может быть выражен разными формулировками.

Token F1 и Jaccard лучше отражают частичное совпадение терминов, однако они также не полностью описывают реальную полезность поискового запроса, потому что разные запросы могут приводить к одинаково релевантной поисковой выдаче.

Поэтому в дальнейшем планируется добавить более содержательную оценку:

```text
generated query → search engine → retrieved links quality
```

или:

```text
generated query → LLM-as-a-judge → quality score
```

---

## Проведённые эксперименты

Было протестировано несколько подходов к генерации запросов:

| Метод | Token F1 | Jaccard | Средняя длина generated query |
|---|---:|---:|---:|
| Simple baseline | 0.1875 | 0.1147 | 10.50 |
| LLM baseline v1 | 0.2566 | 0.1558 | 13.19 |
| Prompt v2 | 0.2406 | 0.1459 | 9.28 |
| Few-shot prompt v3 | 0.2543 | 0.1562 | 11.19 |
| Retrieval few-shot v4 | **0.2683** | **0.1649** | 9.66 |

---

## Описание подходов

### Simple baseline

Простая эвристическая baseline-модель строила поисковый запрос на основе ключевых слов из исследовательского вопроса. Этот подход дал:

```text
Token F1 = 0.1875
Jaccard = 0.1147
```

Такой результат использовался как нижняя граница качества.

### LLM baseline v1

В baseline v1 использовалась LLM с простым prompt, где модели передавались:

```text
research_question
query_id
previous_queries
```

Модель должна была сгенерировать один следующий поисковый запрос. Этот подход повысил качество:

```text
Token F1 = 0.2566
Jaccard = 0.1558
```

### Prompt v2

Во второй версии prompt были добавлены более строгие требования:

- не повторять предыдущие запросы;
- не пересказывать исследовательский вопрос;
- генерировать компактный технический search query;
- выбирать один новый аспект поиска.

Однако результат оказался ниже, чем у v1:

```text
Token F1 = 0.2406
Jaccard = 0.1459
```

Prompt v2 сделал запросы короче, но не улучшил совпадение с эталонными запросами.

### Few-shot prompt v3

В третьей версии prompt были добавлены few-shot-примеры из train-части. Модель получала несколько примеров формата:

```text
research_question + previous_queries → correct next query
```

Результат оказался сопоставим с v1:

```text
Token F1 = 0.2543
Jaccard = 0.1562
```

Few-shot prompt улучшил стиль и длину запросов, но не дал заметного прироста по Token F1.

### Retrieval few-shot v4

В четвёртой версии для каждого validation-примера подбирались релевантные few-shot-примеры из train-части. Подбор осуществлялся по лексическому сходству между текущим примером и обучающими примерами.

Этот подход показал лучший результат:

```text
Token F1 = 0.2683
Jaccard = 0.1649
```

Таким образом, контекстно подобранные few-shot-примеры улучшили генерацию поисковых запросов относительно обычного LLM baseline.

---

## Основные наблюдения

1. Использование LLM заметно улучшает качество по сравнению с простой эвристикой.
2. Слишком строгий prompt не обязательно улучшает качество: v2 оказался хуже v1.
3. Few-shot-примеры помогают модели лучше соблюдать стиль поисковых запросов.
4. Наилучший результат дал retrieval few-shot, где примеры подбирались индивидуально для каждого validation-примера.
5. Текущая постановка ограничена отсутствием `task3`, так как модель не получает контекст уже посещённых источников.
6. Лексические метрики полезны для первичной оценки, но не полностью отражают реальную полезность поисковых запросов.

---

## Текущий вывод

Был реализован baseline генерации поисковых запросов для DeepResearch. Простая эвристика дала `Token F1 = 0.1875`. Использование LLM повысило качество до `Token F1 = 0.2566`.

Лучший результат показал retrieval few-shot prompt, где для каждого validation-примера подбирались релевантные train-примеры:

```text
Token F1 = 0.2683
Jaccard = 0.1649
```

Это показывает, что контекстно подобранные few-shot-примеры улучшают генерацию поисковых запросов.

До предоставления контекста посещённых сайтов был реализован question-only baseline для задачи 4. После появления непустого `train_summerschool_task3.csv` планируется продолжить работу в полной постановке задачи с использованием контекста посещённых источников.
